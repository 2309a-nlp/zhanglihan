# 智能挂号助手 — 系统设计报告

## 1. 系统概述

### 1.1 项目背景

本项目是"人工智能NLP-Agent数字人项目"的医疗智能体-挂号管理工单。目标是用 LLM（大语言模型）替代传统医疗挂号系统的图形表单交互，用户只需用自然语言描述需求（如"帮我大宝挂一个儿科专家的号"），系统即可自动完成意图识别、信息提取、数据库查询与写入。

### 1.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| LLM 推理 | DeepSeek API (deepseek-v4-flash) |
| Agent 框架 | OpenAI 兼容 SDK (Function Calling) |
| 数据库 | SQLite 3 (本地持久化) |
| 前端交互 | Streamlit Web 应用 (端口 8511) |
| 环境管理 | python-dotenv |
| 操作系统 | Windows 11 |

### 1.3 核心能力

- **自然语言意图识别**：理解口语化的挂号/查询/取消需求
- **工具调用（Function Calling）**：LLM 自动选择并调用数据库操作函数
- **多轮对话记忆**：保持上下文，支持"再约一次"等引用式表达
- **多患者管理**：支持本人与家属（大宝、二宝等）的独立档案管理
- **流式响应**：前端实时显示思考过程与工具调用状态
- **排班查询**：查询医生坐诊时间与可用号源

## 4. Agent 核心设计

### 4.1 工具（Tools）定义

系统定义了 8 个工具供 LLM 调用：

| 工具名 | 功能 | 必传参数 | 返回值 |
|--------|------|----------|--------|
| `get_patient_info` | 查询患者档案 | `name` | 患者信息 dict |
| `create_patient` | 新建患者档案 | `name` | `p_ID` |
| `query_department` | 查询科室信息 | `name` | 科室信息 dict |
| `query_doctors` | 查询医生列表 | `dep_id` / `name` | 医生列表/单条 |
| `query_registers` | 查询挂号记录 | `p_id` | 挂号记录列表 |
| `create_register` | 创建挂号 | `dep_id, p_id, reg_time` | `reg_ID` |
| `cancel_register` | 取消挂号 | `reg_id` | 状态 |
| `query_schedule` | 查询坐诊时间 | `doctor_name` / `department` | 排班列表 |

### 4.3 System Prompt 设计

```
你是医院智能挂号助手。请严格使用提供的工具完成用户的挂号、查询、取消等需求。

【核心规则】
1. 必须使用工具：任何数据查询或写入操作都必须调用对应工具，禁止凭空编造。
2. 身份识别：当用户提到"大宝"、"二宝"等家属时，先调用 get_patient_info 查找该患者。
   若不存在，需调用 create_patient 建档。
3. 科室匹配：用户说"儿科专家"时，先 query_department 找科室，
   再 query_doctors 筛选 profession='专家' 的医生。
4. 挂号逻辑：create_register 需要 dep_id, p_id, reg_time。成功后返回 reg_ID 和次序。
5. 取消逻辑：先 query_registers 找到对应记录，获取 reg_ID，再调用 cancel_register。
6. 输出格式：工具调用后，用简洁的中文向用户汇报结果。
7. 坐诊时间：当用户询问某医生的坐诊时间时，使用 query_schedule 工具。
```

设计要点：
- 使用编号规则约束 LLM 行为
- 明确操作先后顺序（先查后写）
- 覆盖边界情况（家属建档、职称筛选）

## 6. 关键流程设计

### 6.1 挂号流程（完整示例）

用户输入：**"帮我大宝挂一个今天下午儿科专家的号"**

```
Step 1: LLM 意图识别
  └── 理解：挂号请求，患者=大宝，科室=儿科，职称=专家，时间=今天下午

Step 2: 调用 get_patient_info(name="大宝")
  └── 结果：不存在 → 需要新建

Step 3: 调用 create_patient(name="大宝")
  └── 结果：p_ID = 7

Step 4: 调用 query_department(name="儿科")
  └── 结果：dep_ID = 1, dep_Name = "儿科"

Step 5: 调用 query_doctors(dep_id=1, profession="专家")
  └── 结果：李儿科 (d_ID=1)

Step 6: 调用 query_schedule(doctor_name="李儿科", date="2026-06-27")
  └── 结果：上午 08:00-12:00 有号源

Step 7: 调用 create_register(dep_id=1, p_id=7, reg_time="2026-06-27 14:00")
  └── 结果：reg_ID = 5, 次序 = 3

Step 8: LLM 生成最终回复
  └── "已为 大宝 成功挂上 儿科专家 李儿科医生，挂号序号 3。"
```

共调用 **6 个工具**，最多 5 轮迭代，实际 2-3 轮即可完成（LLM 可并行多个工具调用）。

### 6.2 取消挂号流程

```
用户："取消我上周挂的消化内科普通号"

Step 1: get_patient_info(name="我") → 需结合上下文识别"我"是谁
        （当前设计假设上一轮对话中已建立患者身份）

Step 2: query_registers(p_id=X, status=1)
  └── 列出该患者所有正常状态的挂号记录

Step 3: LLM 根据"消化内科"筛选出对应 reg_ID

Step 4: cancel_register(reg_ID=Y)
  └── reg_Status = 0

Step 5: 回复用户
```

### 6.3 排班查询流程

```
用户："牙科最近的号哪天的？"

Step 1: query_department(name="牙科")
  └── dep_ID = 2

Step 2: query_schedule(department="牙科")
  └── 返回未来 7 天该科室所有医生的排班

Step 3: LLM 筛选最近的可用号源并格式化输出
```

## 8. 安全与约束设计

### 8.1 安全机制

| 机制 | 实现 | 说明 |
|------|------|------|
| API Key 隔离 | `.env` 文件 | 不硬编码，不提交 Git |
| 参数化查询 | SQLite `?` 占位符 | 防止 SQL 注入 |
| 只读连接 | `get_connection()` | 每次查询后关闭 |
| LLM 约束 | System Prompt | 禁止凭空编造数据 |

### 8.2 数据约束

| 约束 | 实现方式 |
|------|----------|
| 挂号费固定 | 默认 10 元，LLM 可传但实际不动态定价 |
| 排队次序 | `MAX(reg_Order) + 1` 自动计算 |
| 就诊状态 | 枚举：0=取消, 1=正常, 2=完成 |
| 排班可用 | `s_Available = 1` 过滤 |

### 8.3 LLM 调用约束

| 约束 | 实现 |
|------|------|
| 最大迭代次数 | `max_iterations = 5`（防止无限循环） |
| 确定性输出 | `temperature = 0.3`（低随机性，确保工具调用稳定） |
| 工具优先级 | `tool_choice = "auto"`（允许 LLM 判断是否需调用工具） |

## 9. 设计决策与权衡

### 9.1 为什么选择 SQLite 而非 PostgreSQL/MySQL？

| 因素 | 选择 |
|------|------|
| 部署复杂度 | 零配置，单文件数据库，无需安装额外服务 |
| 项目规模 | 实训项目，数据量小（< 1000 条记录） |
| 并发性 | 单用户场景，无高并发需求 |
| 移植性 | 随项目目录整体拷贝即可运行 |

### 9.2 为什么选择 DeepSeek 而非 GPT-4？

| 因素 | 选择 |
|------|------|
| 成本 | 国产模型，API 费用低，适合学生项目 |
| 中文理解 | 中文 NLU 能力优秀，适合医疗场景 |
| Function Calling | 支持 OpenAI 兼容的 tools 协议 |
| 速度 | deepseek-v4-flash 延迟低 |

### 9.3 为什么选择 Streamlit 而非 Flask/FastAPI？

| 因素 | 选择 |
|------|------|
| 开发效率 | 纯 Python，无需 HTML/CSS/JS 三件套 |
| 流式支持 | 原生支持生成器渲染 |
| 部署简便 | `streamlit run` 一键启动 |
| 适用场景 | 内部工具/原型，不需要复杂前端定制 |

### 9.4 为什么用 Function Calling 而非纯 Prompt 工程？

| 方式 | 优点 | 缺点 |
|------|------|------|
| **Function Calling** | 结构化输出，强制类型校验，工具自动发现 | 依赖模型支持 |
| 纯 Prompt + JSON | 兼容所有模型 | 需要额外解析，格式不稳定 |
| ReAct Prompt | 推理+行动框架 | 输出不稳定，需要严格模板 |

本项目选择 Function Calling 因为 DeepSeek 原生支持，且能确保工具调用的准确性和类型安全。

## 10. 扩展性设计

### 10.1 工具扩展

添加新工具只需 3 步：

1. 在 `TOOLS` 列表中追加 JSON Schema 定义
2. 在 `execute_tool()` 中添加 `elif func_name == "new_tool"` 分支
3. 在 `database.py` 中实现对应的 CRUD 函数

LLM 会自动发现新工具并在合适的场景下调用。

### 10.2 数据库扩展

| 扩展方向 | 操作 |
|----------|------|
| 新增表 | 在 `DDL_SCRIPT` 中追加 `CREATE TABLE` |
| 迁移数据 | 使用 `sqlite3` 的 `ALTER TABLE` 命令 |
| 切换数据库 | 实现统一的 `DatabaseBackend` 接口 |

### 10.3 前端扩展

| 扩展方向 | 操作 |
|----------|------|
| 多语言 | 添加 `st.selectbox` 语言切换 |
| 深色模式 | `st.theme` 配置 |
| 数据表格 | 使用 `st.dataframe` 展示历史记录 |
| 文件上传 | 支持批量导入患者 CSV |

## 13. 总结

本项目采用 **LLM + Function Calling + SQLite** 的轻量架构，实现了自然语言驱动的医疗挂号管理。核心设计亮点：

1. **三层架构清晰**：表现层 → 业务逻辑层 → 数据层，职责分明
2. **工具调用标准化**：8 个工具覆盖挂号全流程，Schema 定义严格
3. **流式交互**：用户实时感知 Agent 思考过程，降低等待焦虑
4. **低门槛部署**：纯 Python + SQLite，无需额外服务，双击即可运行
5. **可扩展性强**：新增工具/表/前端功能均可独立扩展

该系统适合作为医疗信息化、NLP Agent、对话式 AI 的教学演示和实训项目。
