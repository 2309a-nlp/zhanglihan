## 三、🔴 待修复问题（按优先级排列）

### P0 — 启动阻断

#### 1. `create_agent` 导入不兼容

- **位置**：`app.py:13` — `from langchain.agents import create_agent`
- **问题**：LangChain v0.3 中 `agents` 模块的顶层 `create_agent` 已被移除/重命名，应使用 `create_openai_tools_agent` 或 `create_tool_calling_agent`
- **症状**：高版本 `langchain>=0.3.0` + `langchain-openai>=0.2.0` 下直接报 `ImportError`
- **修复示例**：

```python
# 删除这行：
from langchain.agents import create_agent

# 改为：
from langchain.agents import create_openai_tools_agent as create_agent
```

---

### P1 — 性能瓶颈

#### 2. Agent 实例每次请求重新创建

- **位置**：`chat()` 路由中每次调用 `agent = create_expense_agent()`
- **问题**：`ChatOpenAI` 模型初始化和提示模板构造开销大，高并发下浪费严重
- **影响**：每轮对话额外增加 200-500ms 延迟
- **建议**：全局缓存 agent 实例

```python
_agent_cache = None

def get_agent():
    global _agent_cache
    if _agent_cache is None:
        _agent_cache = create_expense_agent()
    return _agent_cache
```

#### 3. Debug 模式在生产环境开启

- **位置**：`app.py` 末尾 `app.run(debug=True, ...)`
- **问题**：Flask Debug 模式暴露交互式调试器，存在远程代码执行风险
- **建议**：通过环境变量控制

```python
debug = os.getenv("FLASK_DEBUG", "false").lower() == "true"
app.run(debug=debug, host="127.0.0.1", port=5000)
```

#### 4. 数据库连接未复用

- **位置**：每个数据库操作都新建/关闭连接
- **问题**：每次 `get_db_connection()` → 执行 → `conn.close()` 反复开销
- **建议**：使用 Flask 的 `g` 对象请求内复用

```python
from flask import g

def get_db():
    if 'db' not in g:
        g.db = sqlite3.connect(DB_PATH)
        g.db.row_factory = sqlite3.Row
    return g.db
```

### P2 — 架构与可维护性

#### 5. 单一文件 500+ 行

- **问题**：`app.py` 包含数据库、工具函数、Agent 创建、路由、启动代码全部耦合在一处
- **建议**：按模块拆分

```
工单01/
├── app.py              # Flask 启动和路由（约 80 行）
├── config.py           # 配置、.env 加载、常量
├── database.py         # 数据库连接、CRUD 公共函数
├── tools.py            # LangChain 工具定义
├── agent.py            # Agent 创建、系统提示词
├── .env
├── .env.example
├── .gitignore
├── requirements.txt
├── static/
└── templates/
```

#### 6. 依赖版本未锁定

- **位置**：`requirements.txt` 使用 `>=` 宽松约束
- **问题**：不同时间部署可能安装不兼容版本导致静默失败
- **建议**：生成 `requirements.lock` 锁定版本

```bash
pip freeze > requirements.lock
```

#### 7. `get_all_records` 路由与 `search_transactions` 重复

- **问题**：`/api/records` 路由手动写 SQL 查询，与 `search_transactions` 工具函数的查询逻辑重复
- **建议**：路由内部复用 `search_transactions` 的逻辑，或统一查询入口函数

### P3 — 功能缺失

#### 8. 对话历史仅存内存中

- **位置**：`_chat_histories` 全局字典
- **问题**：服务重启所有历史丢失，且内存占用随用户增多线性增长
- **建议**：存入 SQLite 的 `conversations` 表

```sql
CREATE TABLE IF NOT EXISTS conversations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,
    role TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now', 'localtime'))
);
```

#### 9. 金额/日期输入无校验

- **位置**：`record_expense` / `record_income` 工具函数
- **问题**：负数金额、天文数字、非法日期格式无校验
- **建议**：在写入数据库前增加校验

```python
def _record_transaction(...):
    if amount <= 0:
        raise ValueError("金额必须为正数")
    if amount > 99999999:
        raise ValueError("金额异常，请确认")
```

### P4 — 体验优化

#### 10. 缺少数据可视化

- **问题**：仅文本反馈，无图表直观展示收支结构
- **建议**：前端集成 Chart.js，展示月度收支柱状图 + 分类支出饼图

#### 11. 缺少记录列表视图

- **问题**：用户只能通过聊天查询，无法直接浏览全部数据
- **建议**：新增记录列表面板，支持分页、筛选、排序

#### 12. 功能标签仅过滤快捷按钮，不切换视图

- **位置**：`filterChips()` 函数
- **问题**：点击"支出"标签仅显示 expense 类快捷按钮，对已发送消息无影响
- **建议**：让标签切换时也过滤聊天历史中的相关消息
