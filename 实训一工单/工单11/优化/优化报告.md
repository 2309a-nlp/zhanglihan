## 二、已发现的问题与优化建议

### 问题 1：缺少 Baseline 评估（严重）

训练脚本只在训练后调用 `sequential_evaluator(final_model)`，**没有在训练前对原始 BGE-base 模型做评估**。无法量化微调带来的提升。

**建议**: 在 `train_bge.py` 的模型加载后立即运行一次评估器：

```python
# 训练前评估
print("评估原始模型...")
baseline_results = sequential_evaluator(model)
# 记录到文件
```

如果没有 GPU，可以用 `BAAI/bge-small-en-v1.5` 在 CPU 上做一个简化版评估（取 50 条样本即可获得近似对比）。

---

### 问题 2：缺少 BGE 查询前缀（严重）

BGE 模型（BAAI/bge-series）要求推理和训练时在 query 前添加指令前缀：
- Query: `"Represent this sentence for searching relevant passages: "`
- Document: `"Represent this sentence: "`

当前训练和评估**都未添加**，这会严重降低模型对 query 和 document 的区分度。

`config_sentence_transformers.json` 中 `prompts` 为空：
```json
"prompts": {"document": "", "query": ""}
```

**建议**:
训练时使用 `SentenceTransformer` 的 `prompts` 参数或在 encode 时手动添加上文前缀。评估器的 `InformationRetrievalEvaluator` 也支持设置 `query_prompt` 和 `document_prompt`。

---

### 问题 3：数据预处理缺少质量控制

`prepare_data.py` 只做了列重命名和空值删除。

**缺少的检查**:
- 文本长度过滤（过短/过长的回答降低 embedding 质量）
- 重复数据检查（完全相同的问题或答案对）
- 文本质量过滤（无意义字符、格式错误）

**建议**:

```python
# 在 prepare_data.py 中添加
df = df[df['answer'].str.len() >= 10]      # 过滤过短回答
df = df[df['answer'].str.len() <= 2000]     # 过滤超长回答
df = df[df['question'].str.len() >= 5]     # 过滤过短问题
df = df.drop_duplicates(subset=['question', 'answer'])
```

推荐先做一次**数据探索分析**，打印文本长度分布、常见词汇、空值比例，再决定过滤阈值。

---

### 问题 4：缺少困难负样本挖掘（Hard Negative Mining）

当前使用 `MultipleNegativesRankingLoss`，把 batch 内其他样本的 positive 当作负样本。但随机负样本太容易区分，模型学不到细粒度的语义辨别。

**建议**: 用 BGE 原始模型对全量数据做一次 pre-retrieval，为每个 query 选取最相似但不同配对的 answer 作为 hard negative。Sentence Transformers 5.x 的 `CachedMultipleNegativesRankingLoss` 可动态维护负样本缓存。

如果数据量不大（20K 条），也可以离线构建 hard negatives：用原始 BGE 模型计算所有 query 对所有 answer 的相似度，取 top-k 非匹配项作为 hard negatives。

---

### 问题 5：训练参数可优化

| 参数 | 当前值 | 建议 | 理由 |
|------|--------|------|------|
| Epochs | 1 | 3-5 | Loss 还在下降就停止了，明显欠拟合 |
| Effective batch size | 64 | 128-256 | 对比学习需要更大的 batch 提供更多负样本 |
| eval_steps | 500 | 100 | 当前总步数仅 282，从未触发中间评估 |
| save_steps | 500 | 200 | checkpoint 从未保存 |
| Warmup ratio | 0.1 | 0.05-0.1 | 当前合理，保留 |

**建议**: 将 `num_train_epochs=3`，`eval_steps=50`，`save_steps=100`，并在第一个 epoch 结束时做早停判断。同时减少 `gradient_accumulation_steps` 到 8、增大 `per_device_train_batch_size` 到 8（如果 GPU 显存允许），使有效 batch size 达到 128。

---

### 问题 6：磁盘浪费严重 — 中间文件可清理

| 文件 | 大小 | 是否必要 |
|------|------|----------|
| `checkpoint-282/optimizer.pt` | 831 MB | 训练结束后可删除 |
| `checkpoint-282/model.safetensors` | 418 MB | 与 `bge-finetuned-final/` 重复 |
| `my_train_dataset/` | 12.6 MB | 可从 `data.csv` 重新生成 |
| `my_eval_dataset/` | 1.4 MB | 可从 `data.csv` 重新生成 |

**可清理空间**: **约 1,263 MB**（占总量的 74%）

**建议**: 最终部署时只保留 `bge-finetuned-final/` 目录（约 420 MB）。训练时设置 `save_only_model=True` 可避免保存巨大的优化器状态。

---

### 问题 7：缺少模型推理优化

当前导出为 PyTorch safetensors（418 MB），推理无加速。

**建议**:
- **ONNX 导出**: 用 `optimum` 导出 ONNX，推理速度 2-3x
- **int8 量化**: 用 `bitsandbytes` 量化，模型缩小到 ~100 MB
- **向量索引**: 结合 FAISS IVF 索引，实现大规模向量搜索（当前模型输出已被用来做检索，应搭配适当索引结构）

---

### 问题 8：Matryoshka 维度选择建议

5 个维度性能递减趋势平滑：
- 768d → 256d: Acc@1 仅降 0.95pp（17.10% → 16.15%），存储减少 67%
- 256d → 64d: 降 2.55pp（16.15% → 13.60%），存储再减 75%

**建议**: 
- 256d 是性价比最优维度，适合生产部署
- 可添加 32d 测试极限压缩效果
- 部署时按场景选择：精度优先用 768d，速度优先用 256d
