# 工单04 速度优化报告

## 概述

工单04 在工单03（速度优化+水印处理版）基础上，增加了**熔断器、健康检查、自动恢复、结构化输出**等高稳定性特性，同时继承工单02/03的全部速度优化方案。

---

## 一、问题修复清单

### 1. API Key 硬编码（安全修复）
- **问题**: config.py 中 DEEPSEEK_API_KEY 写死在代码中
- **修复**: 添加 `_load_api_key()` 函数，优先从 `.api_key` 文件读取（Base64编码），回退到环境变量
- **影响**: 提高了密钥安全性，便于 Git 仓库管理

### 2. MySQL 端口错误（连接修复）
- **问题**: config.py 中 MYSQL_PORT=3307，实际运行 MySQL 使用 3306
- **修复**: 修改为 MYSQL_PORT=3306
- **影响**: 数据库连接恢复正常

### 3. httpx SSL 兼容性（兼容修复）
- **问题**: httpx 0.28.1 SSL 握手 bug（_ssl.c:997 EOF），导致 LLM API 调用失败
- **修复**: 将 `httpx.Client` + `OpenAI()` 客户端全线替换为 `urllib.request`
  - `HTTPHandler` + `build_opener` 连接池代替 httpx
  - SSE 流式解析代替 OpenAI 流式 API
  - `urllib.error.URLError/HTTPError` 代替 httpx 异常
- **影响**: LLM 调用稳定，不受 httpx 版本兼容问题影响

### 4. PYTHONPATH 污染（环境修复）
- **问题**: 系统中存在 Python312 site-packages 残余，导致 numpy 等依赖 C ABI 崩溃
- **修复**: app.py 启动时自动清理包含 `Python312` 的路径
- **影响**: 避免了库版本冲突导致的崩溃

### 5. FAISS 索引脏数据（索引修复）
- **问题**: FAISS 索引 2071 向量，DB 2127 条 chunk（少 56 条）
- **原因**: 索引构建时未包含所有文档块
- **修复**: 删除旧索引，重新从 DB 读取全部 chunk 文本 → 生成嵌入向量 → 重建 FAISS 索引
- **影响**: 搜索可覆盖全部文档块，不会遗漏

### 6. Database 健康检查失败（监控修复）
- **问题**: `_run_health_check()` 调用了不存在的 `doc_store.ping()` 和 `reconnect()` 方法
- **修复**: 改为调用存在的 `doc_store.is_healthy()`，并添加 `reconnect()` 方法
- **影响**: 健康检查系统正常运行，自动恢复机制生效

### 7. LLM 预热失败（启动修复）
- **问题**: warmup() 使用 `self._request(messages=..., max_tokens=...)` 传参方式，与 urllib 版 `_request(payload: dict)` 签名不匹配
- **修复**: warmup 改为构建完整 payload dict 调用
- **影响**: 启动时 LLM 连接池成功预热（约 10s）

## 二、速度优化项

### 2.1 混合检索：FAISS + BM25

```
相似度 = α × FAISS语义分 + β × BM25关键词分
α = 0.6, β = 0.4
```

- FAISS 提供语义相似度搜索（512 维 bge-small-zh-v1.5 向量）
- BM25 提供关键词匹配（预构建倒排索引，2127 文档）
- 混合评分弥补各自盲区

### 2.2 HTTP 连接池复用

```python
opener = urllib.request.build_opener(
    urllib.request.HTTPHandler()
)
```
- 首次 LLM 调用时建立连接池（预热约 10s）
- 后续请求复用连接，避免 TCP 握手开销
- 5 秒空闲连接保活检查

### 2.3 语义缓存（SemanticCache）

- 基于语义相似度的缓存（阈值 0.92）
- 相同或相近问题直接返回缓存结果，跳过 LLM 调用
- LRU 淘汰策略（最大 100 条）

### 2.4 启动预热（Warmup）

- LLM 连接池预热（约 10s）
- BM25 索引预加载
- 文本缓存预载到内存

### 2.5 快速模式（Fast Mode）

- LLM_MAX_TOKENS=64（快速模式使用较少 token）
- 检索超时缩短（TOP_K=5, 短文档优先）

## 三、高稳定性特性

| 特性 | 说明 |
|-|-|
| 熔断器 (CircuitBreaker) | 连续 3 次失败→开路→30s 后半开→尝试恢复 |
| 健康检查 (HealthCheck) | 每 60s 检查 DB、FAISS、LLM 状态 |
| 自动恢复 (AutoRecovery) | 异常时自动重置熔断器、重建索引、重连 DB |
| 降级模式 (Downgrade) | LLM 不可用时返回基于检索的 fallback 响应 |
| 连接池保活 (Keepalive) | 每 5s 检查空闲连接，自动重连 |
