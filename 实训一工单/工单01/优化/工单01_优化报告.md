## 1. 严重问题（安全与稳定性）

### 1.1 API Key 硬编码在源码中

**文件**: `config.py`

**问题**: `DEEPSEEK_API_KEY` 直接以字面量写入源代码。当前虽然显示为 `"***"`，但原始代码中曾包含真实密钥。若该文件被提交到 Git、截屏分享或他人查看，API Key 即泄露。

**风险**: 密钥泄露导致第三方盗用 API 额度，产生经济损失。

**修复方案**: 从环境变量读取，并添加 `.env` 文件支持：

```python
import os
from dotenv import load_dotenv

load_dotenv()
DEEPSEEK_API_KEY = os.getenv("DEEPSEEK_API_KEY", "")
if not DEEPSEEK_API_KEY:
    raise ValueError("请设置环境变量 DEEPSEEK_API_KEY")
```

**额外措施**:
- 添加 `.env` 到 `.gitignore`
- 检查 Git 历史，若密钥曾提交过，立刻轮换密钥

### 1.2 MySQL 密码硬编码

**文件**: `config.py`

**问题**: `MYSQL_PASSWORD` 同样硬编码。当前显示为 `os.getenv("MYSQL_PASSWORD", "root")` 但被截断，说明原文件存在编码或内容损坏问题。

**风险**: 同 API Key 泄露风险。此外，密码被截断会导致系统启动即崩溃。

**修复方案**: 统一使用环境变量：

```python
MYSQL_PASSWORD = os.getenv("MYSQL_PASSWORD", "root")
```

### 1.3 配置文件被破坏导致不可运行

**文件**: `config.py`

**问题**: 至少三行存在语法错误或截断：
- `LLM_MAX_TOKENS=***` — Python 语法错误，`***` 不是合法值
- `MYSQL_PASSWORD=os.getenv("MYSQL_PASSWORD", "root")` — 括号不匹配或被截断
- 该类问题会在 `import config` 时直接引发 `SyntaxError`，系统完全不可用

**修复方案**: 逐行检查并修正所有配置项，确保每个值都是合法的 Python 表达式。

## 2. 架构问题

### 2.1 MySQL 硬依赖带来部署门槛

**问题**: 系统强依赖 MySQL 服务（`127.0.0.1:3307`, 数据库 `pdf_qa`）。对于仅需本机使用 PDF 问答的场景，引入外部数据库服务增加了部署复杂度：

- 用户必须先安装并启动 MySQL
- 需手动创建数据库 `pdf_qa`
- 服务宕机会导致整个系统不可用

**建议**: 提供"无 MySQL 模式"：

```
方案 A（推荐）：SQLite 替代
  - 零配置，单文件存储
  - Python 内置，无需额外服务
  - 性能在小数据量（<10 万 chunk）下完全够用
  - 只需修改 DocumentStore 底层实现，接口不变

方案 B：MySQL + SQLite 双模式
  - 环境变量 RAG_DB_BACKEND=mysql|sqlite 切换
  - 高级部署用 MySQL，本机快速测试用 SQLite
```

### 2.2 FAISS 索引与 MySQL 数据一致性风险

**发现**: `vector_store/faiss_index.faiss` 文件大小为 5.8 MB。嵌入维度 512，每向量 4 字节（float32）= 2048 字节，所以约含 2,900 个向量，对应约 2 份招股说明书的文本块。

**问题**: FAISS 文件存在但 MySQL 可能未运行，启动时两者数据可能不一致。若 MySQL 连不上，即使 FAISS 文件存在，BM25 索引也无法重建。

**建议**:
- 启动时检测 FAISS 与 MySQL 数据一致性
- 若 MySQL 不可用，降级为纯 FAISS 语义检索
- 额外用 pickle 持久化文本块列表，减少对 MySQL 的强依赖

### 2.3 `_stream_result` 副作用模式脆弱

**文件**: `qa_engine.py`、`llm_qa.py`

**问题**: 使用生成器 + 实例变量 `_stream_result` 作为"带外"返回值，是典型的多返回值 hack：

```python
for token in engine.chat_stream(query, history):
    yield token
result = engine._stream_result or {}  # 容易忘记读取
```

**风险**:
- 若生成器未完全迭代（异常、提前 break），`_stream_result` 可能为 `None` 或残留旧值
- 非线程安全
- 调用方容易忘记读取该变量

**建议**: 使用自定义结果容器：

```python
class ChunkStreamResult:
    def __init__(self):
        self.answer = ""
        self.source_chunks = []
        self.elapsed_total = 0.0
```

## 3. 代码质量问题

**文件**: `app.py`

**问题**: 渲染参考文档片段的代码在聊天历史回显和输入响应两块中几乎完全相同（约 20 行重复）。

**建议**: 提取为函数：

```python
def _render_source_chunks(source_chunks):
    with st.expander("📎 参考文档片段"):
        for i, src in enumerate(source_chunks, 1):
            score = src.get("score", 0)
            st.markdown(f"**片段 {i}** (相似度: {score:.4f})")
            st.text(src["content"][:200])
            if i < len(source_chunks):
                st.divider()
```

### 3.5 错误处理不足

**多处问题**:
1. `llm_qa.py` 的 `_call_llm_stream` 中，API 异常仅 `print` 并 yield 空字符串，没有重试机制
2. `pdf_processor.py` 的 `process_pdf` 在解析异常时不返回结构化的失败信息
3. `document_store.py` 的 `_ensure_conn` 只做一次重连尝试

**建议**:
- LLM 调用添加指数退避重试（最多 3 次）
- 文档处理使用 `(result, error)` 结构化返回
- 数据库连接失败时提供清晰的用户提示

## 4. 性能优化建议

### 4.1 FAISS 索引应支持增量更新

**现状**: 每次添加或删除文档都完整重建 FAISS 索引（对所有 chunks 重新编码）。

**问题**: 当文档数量增长，重建耗时线性增长。每 3000 个 chunk 约需数秒。

**建议**: FAISS 已提供增量 API，但 `qa_engine._rebuild_faiss` 未使用：

```python
# 增量添加（vector_store.py 已有 add_vectors 但 qa_engine.py 未调用）
self.vector_store.add_vectors(new_ids, new_texts)

# 增量删除（用 remove_ids）
self.index.remove_ids(np.array([remove_id], dtype=np.int64))
```

### 4.2 内存缓存无限增长

**现状**:
1. `_chunk_cache` — 缓存所有 chunk 文本，永不过期
2. `LLMQA.cache` — 查询缓存字典无限增长，键通过 MD5 生成

**建议**: 使用 `cachetools.TTLCache` 替换普通 dict：

```python
from cachetools import TTLCache
self.cache = TTLCache(maxsize=1000, ttl=CACHE_EXPIRE_TIME)
```

### 4.3 嵌入模型预热缺失

**文档宣称**: 技术文档说"嵌入模型预热：初始化时做一次 dummy encode"。

**实际代码**: `vector_store.py` 的 `__init__` 中没有预热逻辑。首次查询会承受模型加载 + 一次推理的延迟。

**建议**: 在 `__init__` 末尾添加：

```python
# 模型预热
_ = self.model.encode(["warmup"], normalize_embeddings=True)
```

### 4.4 BM25 `add_vectors` 中的 TODO

**文件**: `vector_store.py`，`add_vectors()` 方法

**问题**: BM25 增量更新的注释写的是 `# 标记需要从外部重建`，但实际没有做任何重建，传递了空列表 `[]`。这意味着通过 `add_vectors` 添加文档后 BM25 索引是空的。

**建议**: 重构为重建时传入真实文本，或实现增量 BM25 更新。

## 5. 可靠性优化

### 5.1 数据库连接管理

**现状**: `DocumentStore` 每次操作前 `ping() + 重连`，但没有连接池。

**问题**: 高并发下频繁建立/断开 TCP 连接。`ping(reconnect=True)` 可能静默重连到错误的数据库状态。

**建议**: 使用连接池（`SQLAlchemy` 或 `pymysql` + `DBUtils`）。

### 5.2 FAISS 索引持久化不完整

**现状**: 只保存 `.faiss` 文件，未保存 BM25 索引和 chunk_id→text 映射。

**问题**: 启动时必须 MySQL 可用才能重建 BM25，若 MySQL 损坏则 BM25 索引不可恢复，混合检索退化为纯语义检索。

**建议**: 额外持久化：

```python
meta = {"chunk_texts": chunk_texts}
with open(f"{path}.meta.pkl", "wb") as f:
    pickle.dump(meta, f)
```

### 5.3 main.py 的 Streamlit 启动方式

**现状**: 通过 `subprocess.Popen` 静默启动（stdout 和 stderr 全部丢弃）。

**问题**:
- 无法检测启动成功/失败
- stderr 被丢弃导致调试困难
- 重复启动可能创建多个 Streamlit 进程
- 无法优雅关闭

**建议**: 使用 Streamlit CLI API：

```python
import sys
from streamlit.web import cli as stcli
sys.argv = ["streamlit", "run", "app.py"]
sys.exit(stcli.main())
```

### 5.4 Streamlit rerun 导致闪烁

**现状**: 上传/删除文档后调用 `st.rerun()`，触发全页面重载。

**建议**: 用 `st.session_state` 标记需要更新的组件，或使用 `st.empty()` 占位符局部刷新。

## 7. 功能缺陷

### 7.1 不支持扫描件/图片 PDF

**现状**: 仅使用 `pdfplumber` 提取文本，对扫描件（图片型 PDF）完全无效。招股说明书有时是扫描件，当前系统无法处理。

**建议**: 集成 OCR 回退：

```python
try:
    text = page.extract_text()
    if not text.strip():
        from pdf2image import convert_from_bytes
        images = convert_from_bytes(pdf_bytes)
        text = ocr_images(images)  # 调用 PaddleOCR/Tesseract
except:
    pass
```

### 7.2 缺少 PDF 文件校验

**现状**: `st.file_uploader` 限制了 `type="pdf"`，但后端无额外校验。

**建议**: 添加魔数校验：

```python
with open(pdf_path, 'rb') as f:
    if f.read(5) != b'%PDF-':
        return False
```

### 7.3 无流式状态指示

**现状**: 用户输入问题后界面直接等待，若检索耗时 >1s 用户会感觉卡顿。

**建议**: 添加中间状态提示（可选的进度指示）：

```python
with st.spinner("正在检索文档..."):
    # 内部进行检索
    pass
```
